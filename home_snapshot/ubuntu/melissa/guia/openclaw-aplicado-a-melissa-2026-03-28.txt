MELISSA + OPENCLAW — QUE SE TOMO Y QUE QUEDO ACTIVO
Fecha: 2026-03-28

1. IDEA CENTRAL

OpenClaw funciona mejor que muchos agentes "pesados" no por magia, sino por arquitectura:
- separa control, routing y conversación
- evita pedirle todo al prompt
- usa un gateway para decidir comportamiento automáticamente
- mantiene el canal de transporte aparte del cerebro conversacional
- deja que el admin gobierne sin editar código

Eso se aterrizó en Melissa en las piezas siguientes.

2. PATRONES COPIADOS DE OPENCLAW

2.1. Gateway automático
- Melissa ya tiene un gateway estilo OpenClaw.
- Decide capa mínima de comportamiento por rol, canal e intención.
- Archivo principal: /home/ubuntu/melissa/melissa.py
- Clase: TrainerGateway

2.2. Control-plane separado del prompt
- El admin ya no depende solo de "aprender" por prompt.
- Existe un plano de control duro persistente que se aplica antes de enviar.
- Soporta:
  - frases prohibidas
  - arranques prohibidos
  - saludo base
  - segunda burbuja fija
  - tercera burbuja fija
  - cierre fijo
  - fallback fijo cuando Melissa no entiende
  - máximo de burbujas por respuesta
  - reemplazos obligatorios
  - trato al admin
  - reglas de estilo
- Clase: OwnerStyleController

2.3. Router compartido por chat_id
- Melissa ya usa resolución por chat_id para enrutar instancias compartidas.
- Esto permite el patrón "un solo bot token, muchas instancias".
- Se mantuvo y se endureció.

2.4. Cambio de modelo sin reinicio
- Melissa ya permite cambiar modelo por lenguaje natural o comando.
- Modelo actual operativo en clínica:
  - fast: google/gemini-2.5-flash
  - reasoning: google/gemini-2.5-flash

2.5. Capa conversacional separada del transporte
- Telegram y WhatsApp siguen separados del motor de respuesta.
- Typing, burbujas y envío viven aparte del cerebro.
- Esto permite humanizar la entrega sin meter ruido en el LLM.

2.6. Skills y post-filtros
- Melissa ya tiene skills activables, prompt injection y post-process.
- El gateway decide cuándo activarlos sin obligar al admin a recordar IDs.

3. LO QUE QUEDO FUNCIONANDO HOY

3.1. Control duro desde chat admin
Comandos:
- /control
- /control no digas "ay"
- /control no digas "ay", di "entiendo"
- /control no uses claro al inicio
- /control saluda así: Hola, Melissa por acá
- /control segunda burbuja: Estoy lista para ayudarle con la instancia
- /control reset
- /control reset admin
- /control reset paciente

3.2. Control duro desde terminal
Comandos:
- python3 /home/ubuntu/melissa/melissa_cli.py control clinica-de-las-americas
- python3 /home/ubuntu/melissa/melissa_cli.py control clinica-de-las-americas "no digas ay"
- python3 /home/ubuntu/melissa/melissa_cli.py control clinica-de-las-americas "no digas ay, di entiendo"
- python3 /home/ubuntu/melissa/melissa_cli.py control clinica-de-las-americas "saluda así: Hola, Melissa por acá"
- python3 /home/ubuntu/melissa/melissa_cli.py control clinica-de-las-americas "segunda burbuja: Estoy lista para ayudarle con la instancia"
- python3 /home/ubuntu/melissa/melissa_cli.py control clinica-de-las-americas "conmigo háblame de usted y más directo"
- python3 /home/ubuntu/melissa/melissa_cli.py control clinica-de-las-americas "reset admin"
- python3 /home/ubuntu/melissa/melissa_cli.py control clinica-de-las-americas "reset patient"

3.3. Admin vs paciente
- Admin:
  - trato de usted
  - tono más ejecutivo
  - sin emojis
  - no se le habla como paciente
- Paciente:
  - mantiene tono comercial y humano
  - onboarding corto en 2 burbujas
  - mejor respuesta para orientación por foto

3.4. Casos corregidos hoy
- respuesta admin demasiado paciente
- frases flojas tipo "qué te trae por acá"
- reemplazos obligatorios persistentes
- segunda burbuja configurable
- tercera burbuja configurable
- fallback configurable cuando no entiende
- límite de burbujas configurable por admin
- contaminación de estado entre pruebas
- CLI con visibilidad del control duro
- V7 apagado en la base de WhatsApp para volver al generador clásico estable

4. QUE NO SE COPIO DE OPENCLAW A PROPOSITO

- No se copió complejidad innecesaria de tool execution genérica.
- No se convirtió Melissa en un shell agent.
- No se mezcló gobierno del admin con transporte.
- No se metieron abstracciones nuevas donde Melissa ya tenía una capa útil.

5. REGLA OPERATIVA IMPORTANTE

Si el admin corrige tono en lenguaje natural, hay dos capas:
- aprendizaje blando: PromptEvolver
- enforcement duro: OwnerStyleController

La mejora real vino de no depender solo del aprendizaje blando.
