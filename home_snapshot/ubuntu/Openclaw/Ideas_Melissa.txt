Detailed ideas for Melissa OpenClaw-inspired pairing and self-contained skills wrapping.

1) Pairing scaffolding (local, OpenClaw-inspired)
- Objective: zero-ops bootstrap for new Melissa instances with Telegram channel.
- Data:
  - Pending requests: {channel}-pairing.json with entries {chat_id, code, timestamp}
  - Allow list: {channel}-allowFrom.json with allowed chat_ids
- TTL: 1h, max 3 pending per channel
- Code generation: 8 uppercase letters excluding ambiguous ones (O,0,1,I,L) -- using PAIRING_CODE_ALPHABET
- API surface (internal):
  - generate_code_for_chat(chat_id, channel)
  - ensure_pairing_for_chat(chat_id, channel)
  - is_paired(chat_id, channel)
  - approve_code(code, channel, chat_id=None)
  - list_pending(channel)
- Flow:
  - If new chat sends message → check paired; if not, respond with code and instructions and hold processing
  - When approved, chat_id added to allowFrom, and normal flow resumes

2) CLI-like wrapper for pairing (optional)
- pairing list <channel>
- pairing approve <channel> <code> [--chat <chat_id> --notify]

3) Skills wrapping strategy (internal, safe)
- Create internal adapters that map Melissa Skill-IDs to local wrappers that implement the Skill interface
- Allows you to plug in your own skills without copying from external sources
- Keep a manifest of skills in Openclaw folder

4) Documentation cadence and templates
- Create a documented skeleton with sections: Architecture, Data Models, API, Tests, Rollback plan, Security notes

5) Gemini integration plan (fallback if needed)
- Build a separate container/VM for Gemini to operate on the same workspace with restricted network routes
- Provide a small bridge module that feeds Melissa with Gemini-produced results, not as primary path

6) Testing plan
- Unit tests for pairing functions (generate, approve, is_paired)
- Integration tests with webhook mock payloads (Telegram style)
- End-to-end tests for new instance bootstrap

7) Security & governance
- Secrets handling and rotation notes
- Logging and audit trails for pairing approvals
- Access restrictions on Openclaw workspace

8) Rollout plan
- Phase 0: Local prototype in Melissa with Telegram only
- Phase 1: CLI for pairing admin operations
- Phase 2: Multi-channel pairing and Nova integration (if requested later)

9) Risk & rollback
- Add feature flags to disable pairing if required
- Ability to revert to non-paired flow in emergency

10) Evidence & metrics
- Log files placeholder to measure pairing events, approval latency

End of my initial design notes.
